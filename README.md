# Pfaffian-MPS Paper-to-Program Archive

This folder is the GitHub-ready archive for the Pfaffian/Hartree-Fock-Bogoliubov-to-MPS case study. The target task is to reproduce the Pfaffian conversion method from `PhysRevB.105.L081101.pdf` through the `paper-to-program-many-body` workflow.

## Directory Layout

- `Agent-Runs/ClaudeCode/`: ClaudeCode runs, split into `ClaudeCode-Pass/` and `ClaudeCode-Fail/`.
- `Agent-Runs/Codex/`: Codex runs, including self-pairing, cross-spec transfer, and final/meta-spec runs.
- `Agent-Runs/OpenCode/`: OpenCode runs, split into `OpenCode-Pass/` and `OpenCode-Fail/`.
- `Agent-Runs/KimiCode/`: KimiCode runs for Kimi 2.6 self-pairing and final/meta-spec attempts.
- `Specifications/LaTeX-spec/`: generated `latex-spec-LLM-0.tex`, reviewed `latex-spec-LLM-1.tex`, `latex-spec-LLM-1-Final.tex`, `latex-spec-ultra.tex`, and collected specs copied out of agent folders.
- `Prompts/`: all prompt templates used for direct-agent baselines, LLM-0 extraction, LLM-1 review, LLM-2 implementation, PI-Level Review, final-spec generation, and reporting.
- `Skills/`: packaged `paper-to-program-many-body.zip` skill artifact.
- `Summaries/`: experiment summary, pass/fail synthesis, cross-spec transfer analysis, and failure-mode ledger.

## Naming Conventions

`P` means pass and `F` means fail. `CC` means ClaudeCode and `OC` means OpenCode.

Most run folders follow one of two patterns:

- `Agent-Paffian-MPS-LLM1-LLM2-F/P`: implementation from a model-specific spec. In these experiments, `LLM1` is the same model as `LLM0`, and `LLM2` is the implementation model.
- `Agent-Paffian-MPS-Final-LLM2-F/P`: implementation from `Specifications/LaTeX-spec/latex-spec-LLM-1-Final.tex`.

The folder `Agent-Runs/Codex/Codex-Fail/Codex-Paffian-MPS-GPT5.4-F/` is the confirmed GPT 5.4/Codex Phase 1 self-pairing failure; the corresponding final/meta-spec recovery is `Agent-Runs/Codex/Codex-Pass/Codex-Paffian-MPS-Final-GPT5.4-P/`.

## Result Summary

Direct-agent prompting failed as a reliable route. The staged paper-to-program workflow produced 11 passing archived workflow attempts out of 26: 4/11 self-pairing runs, 4/8 cross-spec transfer runs, and 3/7 final/meta-spec runs.

The strongest transfer pattern is asymmetric. Specifications generated by Claude Opus 4.7, DeepSeek V4 Pro, Gemini 3.1 Pro, and Kimi 2.6 all passed when implemented by GPT 5.5/Codex. The reverse direction, using a GPT 5.5 spec with weaker implementation models, failed in all archived cases. The final/meta-spec recovered DeepSeek V4 Pro, GPT 5.4, and Claude Opus 4.7, but not Gemini 3.1 Pro or Kimi 2.6.

For details, start with `Summaries/CODING_EXPERIMENT_SUMMARY.md`.

## Reproducibility Notes

There is no single root-level build system. Run tests and benchmarks from the specific implementation folder being inspected. Most implementations assume Python with NumPy, SciPy, and Matplotlib.

Example commands:

```bash
cd Agent-Runs/Codex/Codex-Pass/Codex-Paffian-MPS-GPT5.5-P
python3 tests/tests.py
python3 scripts/benchmark.py
```

The export excludes local virtual environments, Python bytecode caches, Claude local settings, `.DS_Store` files, LaTeX auxiliary build products, and duplicated run-local copies of `PhysRevB.105.L081101.pdf`. Use the canonical source paper from the main workspace `References/` folder or the journal source when reproducing the workflow.
